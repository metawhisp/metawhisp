# WAL Protocol

## Назначение
WAL — это save-file состояния сессии. Его задача — дать следующей сессии продолжить без потери качества. Историю хранит git, не WAL.

## Когда обновлять
1. **В конце каждой сессии — без исключений.** Если сессия крашнулась (контекст, ошибка) — человек правит WAL вручную. WAL не должен описывать несуществующее состояние.
2. **Перед деструктивным рефакторингом** — чекпойнт.
3. **При переключении контекста** между модулями — обновить WAL текущего модуля до переключения.

## Дисциплина размера
- WAL ≤ ~3000 токенов
- Завершённые задачи COLLAPSE: 5 строк деталей → 1 строка "COMPLETE"
- Детали живут в git, не в WAL
- WAL распух = слишком много открытых тредов = нужно приоритизировать

## Обязательные секции
- **Current Phase** — активный модуль/спек
- **Completed** — коллапсированные one-liner'ы
- **In Progress** — DONE/TODO с spec:// и путями файлов
- **Known Issues** — пронумерованно, с локацией
- **Decisions Pending** — на обсуждение с человеком
- **Session Context** — с чего начать, ключевые файлы, что запустить, анти-инструкции

## Анти-инструкции
Обязательны строки "Watch out: НЕ ТРОГАЙ X". Защищают стабильный код от типичного AI поведения: "вижу функцию, её можно улучшить" → "улучшение" ломает работающее.

## Тест полноты
Может ли следующая сессия продолжить без внешнего контекста? Да — WAL хороший. Нет — добавь недостающее.

## WAL и человек
AI пишет, человек верифицирует. AI может написать "COMPLETE" когда edge cases не покрыты. Утренний ритуал человека:
- Прочитать WAL
- Соответствует ли он тому, что помнишь?
- Нет — поправить до запуска новой сессии
- Голова человека (стабильная память) бьёт WAL (volatile) при расхождении

## Восстановление после краха
1. `git log` + `git diff` — что случилось
2. Обновить WAL вручную из памяти + git
3. Это и делает человека незаменимым — living WAL backup
