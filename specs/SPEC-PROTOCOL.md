# Spec Protocol

## Иерархия разрешения конфликтов (Memory Fence)

Приоритет — выше выигрывает:
1. **Человек** бьёт спеку (человек меняет спеку)
2. **Спека** бьёт код (код обязан соответствовать спеке)
3. **Тесты** — executable spec (конфликт теста и спеки = баг в одном из них)

Без явных правил приоритета два процесса создают data race.

## Когда AI не согласен со спекой

1. Реализуй ровно как написано (спека выигрывает)
2. Добавь REVIEW-маркер:
   `<!-- REVIEW: §X.Y возможно выгоднее [альтернатива], потому что [причина] -->`
3. Отчитайся в конце сессии
4. Человек решает в следующем цикле

Цена: три строки текста, секунды на запись, минута на чтение.
Цена несоблюдения: тихие изменения → каскадные несоответствия.

## Формат файлов спек

### Именование
- `PROP-XXXX.md` — протокол/техническая спека (интерфейсы, контракты)
- `FEAT-XXXX.md` — спека фичи (пользовательский сценарий)

### Обязательная структура
- Иерархические якоря: `{#section.subsection}`
- URI-ссылки: `spec://<module>/<doc>#<section>`
- Обоснование для неочевидных решений
- Changelog в конце

### Якоря
```markdown
# FEAT-0001: Название {#root}

## 1. Секция {#section}

### 1.1. Подсекция {#section.subsection}
```
Точка в якоре = иерархия.

### Changelog
```markdown
## Changelog
- [2026-04-16] §3.1: mic+system audio mix — решил проблему с голосом пользователя
- [2026-04-16] §2.4: hallucination filter переиспользует static из coordinator
```

## Формат коммитов

```
[module] описание изменения

Implements: spec://module/SPEC-ID#section.subsection
- Деталь 1
- Деталь 2
```

Один коммит = одно логическое изменение = одна секция спеки + код + (если применимо) тесты.

## Размещение критической информации

Из-за эффекта "Lost in the Middle":
- Ограничения, критерии приёмки, инварианты → первые параграфы спеки
- Или в финальной секции "Invariants"
- НИКОГДА не хоронить в середине длинного документа
