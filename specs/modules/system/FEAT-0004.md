# FEAT-0004: Permissions Service {#root}

## User story {#story}

**Как** пользователь,
**Я хочу** что когда я включаю фичу в Settings (Meeting Recording / Screen Context) — macOS диалог разрешения появляется СРАЗУ, а не молча игнорируется.

## Acceptance criteria (Critical) {#acceptance}

- Активный триггер TCC диалогов (не надеемся на Info.plist)
- Один сервис-фасад над всеми разрешениями
- На `applicationDidBecomeActive` пермишны перечитываются (после System Settings → возврат в app)
- Если разрешение денайнуто — автоматически открывается System Settings
- UI показывает актуальный статус (не кэширует устаревшее)

## 1. Зачем этот сервис {#rationale}

macOS TCC работает так: `NSScreenCaptureUsageDescription` в Info.plist — это **только текст** в диалоге. Для показа диалога нужно **активно** вызвать API:

- **`CGRequestScreenCaptureAccess()`** — триггерит старый TCC Screen Recording диалог
- **`SCShareableContent.excludingDesktopWindows(...)`** — триггерит **отдельный** ScreenCaptureKit диалог (macOS 14+)

С macOS 14+ это **две разные TCC записи**. Без обоих вызовов фичи работать не будут даже если Info.plist правильный.

В референсе это `ScreenCaptureService.requestAllScreenCapturePermissions()`. У нас — `PermissionsService`.

## 2. API {#api}

```swift
@MainActor
final class PermissionsService: ObservableObject {
    static let shared = PermissionsService()

    @Published var screenRecordingGranted: Bool
    @Published var microphoneGranted: Bool
    @Published var accessibilityGranted: Bool

    func refresh()
    var hasScreenRecording: Bool
    func requestScreenRecording() async -> Bool
    func openScreenRecordingSettings()
    func requestMicrophone() async -> Bool
    func openMicrophoneSettings()
    func requestAccessibility()
}
```

**Файл:** `Services/System/PermissionsService.swift`

## 3. Триггеры {#triggers}

### 3.1. При включении фичи в Settings {#triggers.settings-toggle}
`AppDelegate.observeIntelligenceSettings()` слушает `UserDefaults.didChangeNotification`. При переходе `screenContextEnabled: false → true` или `meetingRecordingEnabled: false → true` вызывает `PermissionsService.requestScreenRecording()`.

Пользователь видит диалог сразу после toggle, а не позже когда первый раз нажмёт Record.

### 3.2. При старте фичи {#triggers.feature-start}
`SystemAudioCaptureService.start()` и `ScreenContextService.startMonitoring()` дополнительно проверяют и триггерят — на случай если permission отозвали между toggle и Record.

### 3.3. На переход app в foreground {#triggers.app-activation}
`AppDelegate.applicationDidBecomeActive` → `PermissionsService.refresh()`. Когда пользователь гранит разрешение в System Settings и возвращается в MetaWhisp — статус обновляется без перезапуска.

## 4. Accessibility специфика {#accessibility-quirk}

**Известная проблема:** после rebuild .app с новой подписью — TCC иногда не авторегистрирует его в Accessibility списке. Требует ручного добавления через `+` в System Settings → Privacy & Security → Accessibility.

Причина: Accessibility TCC привязывается к точному code signature hash. Ad-hoc подпись меняется на каждом rebuild (у нас нет Apple Developer ID).

**Workaround:** `AXIsProcessTrustedWithOptions([kAXTrustedCheckOptionPrompt: true])` — триггерит системный диалог "Allow MetaWhisp?". Иногда срабатывает, иногда нет (зависит от стабильности Launch Services).
