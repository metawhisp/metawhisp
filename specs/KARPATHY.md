# Karpathy Principles — обязательный базовый режим работы

**Source:** https://github.com/forrestchang/andrej-karpathy-skills

**Критически важно:** эти 4 принципа применяются ко ВСЕМ дальнейшим изменениям в проекте MetaWhisp. Нарушение — баг поведения, не стиль.

---

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Перед реализацией:
- Явно озвучить предположения. Если неуверен — **спросить**.
- Если есть несколько вариантов трактовки — **представить их**, не выбирать молча.
- Если проще — **сказать об этом**. Push back когда уместно.
- Если неясно — **стоп**. Назвать что непонятно. Спросить.

**Test:** спрашиваю ли я до кода, или догадываюсь и правлю после ошибок?

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- Никаких фич сверх запрошенного.
- Никаких абстракций для кода используемого 1 раз.
- Никакой "гибкости" которую не просили.
- Никакого error handling для невозможных сценариев.
- Если 200 строк можно заменить на 50 — переписать.

**Test:** senior engineer сказал бы "overcomplicated"? Если да — упростить.

**Applied to specs:** если пишу SPEC на 300 строк чтобы починить проблему "повторы советов" — я overengineer. 20-строчный фикс — правильный путь.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

При редактировании:
- Не "улучшать" соседний код, комменты, форматирование.
- Не рефакторить то что не сломано.
- Соблюдать существующий стиль даже если сделал бы иначе.
- Увидел dead code — **упомянуть**, не удалять.

Когда мои изменения создают orphans:
- Удалить imports/variables/functions которые **мои** изменения оставили неиспользуемыми.
- **НЕ удалять** pre-existing dead code без явного запроса.

**Test:** каждая изменённая строка должна прямо следовать из запроса юзера?

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Превратить задачу в проверяемую цель:
- "Добавь валидацию" → "Напиши тесты для невалидных входов, заставь их pass"
- "Почини баг" → "Напиши тест воспроизводящий баг, заставь его pass"
- "Отрефакторь X" → "Убедись что тесты проходят до и после"

Для multi-step — краткий план:
```
1. [Шаг] → verify: [проверка]
2. [Шаг] → verify: [проверка]
3. [Шаг] → verify: [проверка]
```

Сильные критерии → могу loop сам без уточнений. Слабые ("make it work") → постоянно спрашиваю.

## MetaWhisp-specific application

### Приложение уже работает
Default режим — **минимальные хирургические изменения**. Не переписывать архитектуру. Не городить ассистентов. Не добавлять abstraction layers. Пока не доказано что минимум не работает.

### Когда нужны "глобальные изменения"
**Только** в этих случаях (не раньше):

1. **Small fix не сдвигает метрики 2-3 попытки подряд**
   Например: прогноз "увеличь окно с 5 до 50 починит повторы" → измерили → не работает → ещё одна попытка с no_advice escape → не работает → **тогда** обсуждаем более крупное изменение.

2. **Пользователь запрашивает принципиально новую функцию**
   Когда функции нет в коде и её нельзя сделать surgical-правкой (например "добавь BLE поддержку для Omi wearable" — это новый AudioSource, неизбежно новые файлы).

3. **Изменение контракта требует правок в нескольких слоях**
   Пример: изменить формат транскрипции с `String` на `[Segment]` — ломает API coordinator + views + history + processing. Это global change, требует plan.

4. **Privacy / security инвариант под угрозой**
   Любое изменение которое может сделать так что пользовательские данные уходят куда-то куда не должны → обязательно обсуждение.

### Запрет на speculative architecture
- **Нельзя** писать docs-на-300-строк про "как бы могло быть" до подтверждения что короткий фикс не работает.
- **Нельзя** добавлять модели/сервисы/endpoints "на всякий случай" или "для будущих ассистентов".
- **Нельзя** копировать архитектуру референса (Omi) целиком — берём только минимальные paterns которые **прямо сейчас** решают конкретную задачу.

### Успех
Мы работаем по Карпати правильно если:
- В diff'ах **только прошенные изменения**
- Код **прост с первого раза**, без переписываний из-за overcomplicated
- **Вопросы до реализации**, не после ошибок
- **Чистые минимальные PR**, без drive-by "улучшений"
